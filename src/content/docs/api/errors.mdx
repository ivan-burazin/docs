Daytona uses conventional HTTP response codes to indicate the success or failure of an API request.

## HTTP Status Codes

| Code | Description |
|------|-------------|
| `200` | OK - Request succeeded |
| `201` | Created - Resource was successfully created |
| `204` | No Content - Request succeeded with no response body |
| `400` | Bad Request - Invalid request format or parameters |
| `401` | Unauthorized - No valid API key provided |
| `403` | Forbidden - Valid API key but insufficient permissions |
| `404` | Not Found - The requested resource doesn't exist |
| `429` | Too Many Requests - Rate limit exceeded |
| `500` | Server Error - Something went wrong on our end |
| `502` | Bad Gateway - Invalid response from upstream server |
| `503` | Service Unavailable - Server is temporarily unavailable |
| `504` | Gateway Timeout - Request timed out |

## Error Response Format

When an error occurs, the API returns a JSON response with error details:

```json
{
  "error": {
    "code": "invalid_request",
    "message": "The provided API key is invalid",
    "status": 401,
    "details": {
      "field": "Authorization",
      "reason": "Missing or invalid bearer token"
    }
  }
}
```

### Error Object Properties

| Property | Type | Description |
|----------|------|-------------|
| `code` | string | Machine-readable error code |
| `message` | string | Human-readable error message |
| `status` | number | HTTP status code |
| `details` | object | Additional error context (optional) |

## Common Error Codes

| Error Code | Description |
|------------|-------------|
| `invalid_request` | The request was malformed or missing required parameters |
| `authentication_required` | No API key was provided |
| `invalid_auth` | The provided API key is invalid |
| `permission_denied` | The API key doesn't have permission for this action |
| `not_found` | The requested resource doesn't exist |
| `rate_limit_exceeded` | Too many requests in a given time period |
| `server_error` | An unexpected error occurred on our servers |

## Handling Errors

Here's an example of proper error handling in different languages:

### cURL
```bash
curl -i -H "Authorization: Bearer YOUR_API_KEY" \
  https://api.daytona.io/v1/build/invalid-id
```

### Python
```python
import requests

try:
    response = requests.get(
        'https://api.daytona.io/v1/build/invalid-id',
        headers={'Authorization': f'Bearer {API_KEY}'}
    )
    response.raise_for_status()
except requests.exceptions.HTTPError as e:
    error = e.response.json()
    print(f"Error {error['error']['status']}: {error['error']['message']}")
```

### JavaScript
```javascript
try {
  const response = await fetch('https://api.daytona.io/v1/build/invalid-id', {
    headers: {
      'Authorization': `Bearer ${API_KEY}`
    }
  });
  if (!response.ok) {
    const error = await response.json();
    throw new Error(`${error.error.status}: ${error.error.message}`);
  }
} catch (error) {
  console.error('API Error:', error.message);
}
```

## Rate Limiting

When you exceed the rate limit, you'll receive a `429 Too Many Requests` response. The response headers will include:

```http
X-RateLimit-Limit: 5000
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1623456789
```

See the [Rate Limiting](/api/pagination#rate-limiting) section for more details.
